Proposal: Reliability, Ingest, and RAG Quality (Elevator Pitch)
==============================================================

The problem
-----------
Teams hit our API from their own GCP spaces and see flaky uploads
(often the first attempt fails), slow small uploads, and outright
failures on large files.

What we are changing
---------------------
**1. Stop tying “receive file” to “chunk and index” on one HTTP
request.** Today the full file hits the API and heavy work runs in
that request path. For large inputs that couples network transfer,
timeouts, memory, and model cold start. The move is: **land the file
in GCS first** (signed URL or resumable upload), then **async**:
worker reads from GCS (stream or bounded buffering), runs the **same**
chunking → embed → index pipeline, and exposes **job status** for
polling. That removes API body bottlenecks for big binaries and
matches how object storage and workers are meant to work.

**2. Warmup and retries.** Add an explicit **warmup**
path (endpoint, readiness that reflects real readiness, optional
background warmers). Product teams can also ping us once at their app
startup. Pair that with a **retry protocol**: idempotent init/finalize,
backoff with jitter, dedupe on finalize—so transient first-try
failures don’t become user-visible failures.

**3. Async ingestion as a first-class pipeline.** After GCS reports
completion (or via finalize events), enqueue work: **queue +
event-driven** triggers, idempotent processing, dead-letter/retry for
failed jobs. The API stays a **thin control plane** (auth, metadata,
issuing upload sessions, status); **ingest** owns heavy CPU and I/O.
Over time we split **chat** and **ingest** into separate services so
latency-sensitive chat isn’t competing with bulk indexing.

**4. Chat: faster warm path.** Profile the chat pipeline, parallelize
independent steps where safe, add light caching where it helps—without
overbuilding budgets in this doc.

Outcome
-------
More reliable uploads (especially large ones), clearer separation
between moving bytes and processing them, better operational story for
ingest, and answers that only cite sources when retrieval is actually
confident enough to support them.
