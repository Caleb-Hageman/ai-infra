Implementation Plan: Upload Reliability, Latency, and RAG Guardrails
=====================================================================

Goal
----
Reduce upload failures/latency, support large files reliably, and
prevent low-confidence RAG answers with misleading citations.


Current architecture (as understood)
--------------------------------------
- Client sends the **full file** to the API.
- The API runs **server-side AI chunking** (e.g. Hugging Face–based)
  on that payload, then **persists chunks** (and downstream embed/index
  steps as today).

Implications for the optimal solution
-------------------------------------
- **Chunking stays server-side.** Client-side “split into N KB chunks”
  does **not** replace semantic/AI chunking; the optimal path is still
  **get the full object into durable storage (GCS) reliably**, then run
  the **same** chunking + embed + index pipeline in a **worker**, not on
  the synchronous upload HTTP request.
- **Do not chain upload + HF chunking + embed in one request** for
  large files: timeouts, memory, and cold model load dominate. After GCS
  landing, enqueue a job that **downloads (or streams) from GCS → runs
  existing chunker → stores chunks**.
- **Parallel upload of many client chunks** is a poor fit here (chunks
  are defined by the model, not fixed byte ranges). Prefer **GCS
  resumable/multipart for the single file**, then one **ingestion job**
  that owns chunking.
- **Warmup** should include whatever the ingest path needs (HF
  tokenizer/model load, first chunk pass) if those are cold-start
  contributors.
- **Worker sizing**: large files + HF inference may need more RAM/CPU
  (or a dedicated ingest runtime) than the thin API.


Feature 1: Upload Warmup + Retry Protocol
-----------------------------------------
Task 1.1: Implement explicit warmup flow
- Subtask: Add `GET /warmup` endpoint that initializes DB client,
  storage client, and any lazy dependencies.
- Subtask: Add startup probe that only reports ready after warmup
  checks pass.
- Subtask: Add background warmers (scheduled/internal) that call
  warmup endpoint periodically.
- Note: Product teams can also send a lightweight init request from
  their process startup to warm paths on their side; no server change
  required for that pattern.

Task 1.2: Add client/server retry protocol for transient failures
- Subtask: Define retry-safe upload lifecycle endpoints (init/finalize)
  with idempotency keys.
- Subtask: Implement exponential backoff + jitter policy on retryable
  failures.
- Subtask: Add dedupe behavior so repeated finalize calls do not
  duplicate records.


Feature 2: Large File Uploads via Direct-to-GCS
------------------------------------------------
Task 2.1: Move transfer path off API body handling
- Subtask: API issues signed URL or resumable upload session.
- Subtask: Client uploads file directly to GCS (not proxied through
  API).
- Subtask: API receives completion callback/event and marks upload
  state.
- Subtask: **Ingest worker** reads the GCS object and runs the
  **existing AI chunking** step (then embed/index as today); behavior
  matches current semantics, only the **trigger and placement** move off
  the upload request.

Task 2.2: Use native multipart/resumable APIs for large files
- Subtask: Define size threshold for resumable/multipart path.
- Subtask: Enable resume support after interrupted uploads.
- Subtask: Validate object integrity with checksum before processing.

Task 2.3: Keep large-file reliability controls
- Subtask: Supported content type policy (as needed).
- Subtask: Track upload session states (`initiated`, `uploading`,
  `uploaded`, `finalized`, `failed`).
- Subtask: Add cleanup for abandoned/incomplete sessions.

Task 2.4: Chunking pipeline placement
- Subtask: Refactor so HF (or equivalent) chunking runs in
  **ingest/async** context, reading from GCS (stream or bounded buffer),
  not inline on multipart upload handler.
- Subtask: Preserve chunk boundaries and storage format compatible with
  current RAG index expectations.


Feature 3: Fast Control Plane + Heavy Data Plane Split
------------------------------------------------------
Task 3.1: Split API responsibilities
- Subtask: Control plane handles auth, metadata, signed URL/session
  issuance, and status reads.
- Subtask: Data plane handles **AI chunking**, embedding, indexing, and
  storage writes (heavy path today; should not sit on upload HTTP thread
  for large inputs).
- Subtask: Return immediate job/session IDs for async progress tracking.

Task 3.2: Separate services by workload
- Subtask: Create/assign `chat-service` for low-latency request/response
  paths.
- Subtask: Create/assign `ingest-service` for long-running
  ingestion/indexing jobs.
- Subtask: Ensure independent scaling/operational ownership per service.


Feature 4: Async Ingestion Pipeline
-----------------------------------
Task 4.1: Queue-based ingestion
- Subtask: On upload complete (GCS object ready), publish ingestion job
  message.
- Subtask: Worker consumes message and runs **AI chunking (HF path) →
  embed → index** pipeline on the stored object.
- Subtask: Persist job status and error details for client polling.

Task 4.2: Event-driven processing
- Subtask: Use GCS object finalize event to trigger ingestion job
  creation.
- Subtask: Ensure idempotent event handling (same event does not
  double-process).
- Subtask: Add dead-letter/retry path for failed jobs.


Feature 5: Chat Runtime Execution Optimization
----------------------------------------------
Task 5.1: Reduce warm-path latency
- Subtask: Profile and time each stage (retrieval, rerank, model call,
  response formatting).
- Subtask: Parallelize independent I/O calls where safe.
- Subtask: Add lightweight caching for repeated retrieval/model
  sub-results where applicable.


Feature 6: RAG Low-Retrieval Hallucination Guardrails
------------------------------------------------------
Task 6.1: Retrieval confidence gates
- Subtask: Add `top1_similarity_min` threshold.
- Subtask: Add `citation_similarity_min` and minimum count of valid
  citations.
- Subtask: If thresholds fail, force abstention response.

Task 6.2: Abstention and citation policy
- Subtask: Return generic "insufficient indexed information" response
  when confidence is low.
- Subtask: Do not include citations below threshold.
- Subtask: Disable "what I can answer instead" fallback for
  low-confidence retrievals.

Task 6.3: Two-stage retrieval quality control
- Subtask: Retrieve broad candidate set.
- Subtask: Rerank and apply strict cutoff before generation context
  assembly.
- Subtask: Only allow claims backed by retained citations.

Task 6.4: Validation rollout
- Subtask: Run shadow mode (log thresholds, no behavior change) first.
- Subtask: Compare abstention rate vs incorrect-answer rate on
  evaluation set.
- Subtask: Enable hard gating after threshold calibration.


Delegation Order (Suggested)
----------------------------
1) Feature 6 (RAG guardrails shadow mode)
2) Feature 1 (warmup/retry) + Feature 2 (direct-to-GCS upload path)
3) Feature 4 (async ingestion + event-driven triggers)
4) Feature 3 (service split)
5) Feature 5 (chat runtime optimization)


Definition of Done (Per Feature)
--------------------------------

- Before/after spot-check on latency/failure for the touched flows.
